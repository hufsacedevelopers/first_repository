# 기획 배경

## 문제 정의
- 장애인 및 청년 구직자는 기업의 실제 근무 환경 정보를 얻기 어렵습니다.
- 기업은 장애인 채용 시 활용 가능한 지원금과 제도를 충분히 인지하지 못합니다.
- 기존 채용 플랫폼은 공고 중심으로 정보가 제공되어 의사결정을 돕기 어렵습니다.

## MVP 운영 범위 (1차)
- 1차 MVP는 **경기도 중심(지역 집중 전략)**으로 운영합니다.
- 전국 확장은 구조적으로 가능하게 유지하되, 초기 검증은 경기도 데이터/사용자 시나리오에 집중합니다.
- 목적:
  - 데이터 품질 관리 난이도 축소
  - 실제 사용자 여정 완성 속도 향상
  - 발표/데모 메시지 명확화(경기도 장애인 취업 의사결정 MVP)

## 경기도 집중 전략의 합리적 이유
1. **데이터 일관성 확보**
   - 초기 단계에서 전국 데이터를 동시에 맞추면 필드 불일치/결측 대응 비용이 커집니다.
   - 경기도 중심으로 시작하면 데이터 정규화 규칙을 안정적으로 확립할 수 있습니다.
2. **개발 범위 통제**
   - MVP의 핵심은 기능 완성(탐색 -> 확인 -> 행동)이지 커버리지 확장이 아닙니다.
   - 지역 범위를 축소하면 상세 페이지, 지원 흐름, 추천 경험을 더 빠르게 완성할 수 있습니다.
3. **검증 가능한 사용자 시나리오**
   - 특정 지역(경기도)으로 한정하면 사용자 인터뷰/피드백 대상을 명확히 잡을 수 있습니다.
   - KPI(조회 -> 클릭 -> 상세진입 -> 행동전환) 추적이 쉬워집니다.
4. **확장 가능한 구조 유지**
   - API/스키마는 전국 확장 가능 형태를 유지하므로, 경기도 MVP 성공 후 수도권/전국 확장 리스크가 낮습니다.

## 문서/운영 시 고려사항
- **서비스 범위 명시**
  - README, 발표자료, 화면 내 안내 문구에 "MVP 1차: 경기도 중심"을 명확히 표기합니다.
- **확장 계획 동시 제시**
  - "경기도 -> 수도권 -> 전국" 단계적 확장 계획을 함께 제시해 범위 축소 오해를 줄입니다.
- **지표 기반 의사결정**
  - 경기도 MVP에서 아래 지표를 수집해 확장 시점을 판단합니다:
    - 추천 페이지 진입률
    - 상세 페이지 전환율
    - 지원금 계산기 사용률
    - 재방문율
- **리스크 관리**
  - 특정 지역 집중으로 인한 대표성 논란을 줄이기 위해, 데이터/기능 확장 로드맵을 정기 공유합니다.
- **기술 설계 원칙**
  - region 하드코딩 최소화(기본값만 경기)
  - 필터/쿼리 파라미터 기반으로 전국 확장 가능 구조 유지

## 현재 제품 상태 진단
현재 서비스는 메인 화면에서 기업 카드와 추천 일자리 카드를 탐색할 수 있지만,  
"자세히 보기" 이후 행동으로 이어지는 흐름(지원, 저장, 신청)이 약합니다.

핵심 갭:
- 탐색은 가능하지만 "확인 -> 행동" 단계가 끊김
- 장애 유형 관점 필터가 부족함
- 지원금 계산 결과가 실제 신청 행동으로 연결되지 않음

## 사용자별 주요 니즈와 갭

- 원하는 것:
  - 내 장애 유형/근무 조건에 맞는 기업과 공고 탐색

  - 관심 공고 저장 및 재방문

  - 장애 유형 필터 부재
  - 회사/공고 상세 페이지 부족
  - 찜/북마크 기능 부재

#### 버그 수정 내역

**[백엔드] `live_job_service.py` resultCode 조건 오류**
- 위치: `backend/app/services/live_job_service.py:48`
- 증상: `/jobs/live`, `/jobs/live-with-env` 항상 502 반환
- 원인: 외부 API(data.go.kr)가 성공 코드로 `"0000"`을 반환하는데 코드가 `"00"`만 허용
- 수정: `result_code != "00"` → `result_code not in ("00", "0000")`

**[프론트] Next.js 16 `params` async 미처리**
- 위치: `app/job/[id]/page.tsx`, `app/company/[id]/page.tsx`
- 증상: 상세 페이지 전부 404
- 원인: Next.js 15+에서 `params`가 `Promise` 타입이 됐는데 `await` 없이 접근 → `id`가 `undefined` → `parseInt(undefined)` = `NaN` → `getJobById` 즉시 `null` 반환
- 수정: `params: { id: string }` → `params: Promise<{ id: string }>`, `const { id } = await params`
