# 기획 배경

## 문제 정의
- 장애인 및 청년 구직자는 기업의 실제 근무 환경 정보를 얻기 어렵습니다.
- 기업은 장애인 채용 시 활용 가능한 지원금과 제도를 충분히 인지하지 못합니다.
- 기존 채용 플랫폼은 공고 중심으로 정보가 제공되어 의사결정을 돕기 어렵습니다.

## MVP 운영 범위 (1차)
- 1차 MVP는 **경기도 중심(지역 집중 전략)**으로 운영합니다.
- 전국 확장은 구조적으로 가능하게 유지하되, 초기 검증은 경기도 데이터/사용자 시나리오에 집중합니다.
- 목적:
  - 데이터 품질 관리 난이도 축소
  - 실제 사용자 여정 완성 속도 향상
  - 발표/데모 메시지 명확화(경기도 장애인 취업 의사결정 MVP)

## 경기도 집중 전략의 합리적 이유
1. **데이터 일관성 확보**
   - 초기 단계에서 전국 데이터를 동시에 맞추면 필드 불일치/결측 대응 비용이 커집니다.
   - 경기도 중심으로 시작하면 데이터 정규화 규칙을 안정적으로 확립할 수 있습니다.
2. **개발 범위 통제**
   - MVP의 핵심은 기능 완성(탐색 -> 확인 -> 행동)이지 커버리지 확장이 아닙니다.
   - 지역 범위를 축소하면 상세 페이지, 지원 흐름, 추천 경험을 더 빠르게 완성할 수 있습니다.
3. **검증 가능한 사용자 시나리오**
   - 특정 지역(경기도)으로 한정하면 사용자 인터뷰/피드백 대상을 명확히 잡을 수 있습니다.
   - KPI(조회 -> 클릭 -> 상세진입 -> 행동전환) 추적이 쉬워집니다.
4. **확장 가능한 구조 유지**
   - API/스키마는 전국 확장 가능 형태를 유지하므로, 경기도 MVP 성공 후 수도권/전국 확장 리스크가 낮습니다.

## 문서/운영 시 고려사항
- **서비스 범위 명시**
  - README, 발표자료, 화면 내 안내 문구에 "MVP 1차: 경기도 중심"을 명확히 표기합니다.
- **확장 계획 동시 제시**
  - "경기도 -> 수도권 -> 전국" 단계적 확장 계획을 함께 제시해 범위 축소 오해를 줄입니다.
- **지표 기반 의사결정**
  - 경기도 MVP에서 아래 지표를 수집해 확장 시점을 판단합니다:
    - 추천 페이지 진입률
    - 상세 페이지 전환율
    - 지원금 계산기 사용률
    - 재방문율
- **리스크 관리**
  - 특정 지역 집중으로 인한 대표성 논란을 줄이기 위해, 데이터/기능 확장 로드맵을 정기 공유합니다.
- **기술 설계 원칙**
  - region 하드코딩 최소화(기본값만 경기)
  - 필터/쿼리 파라미터 기반으로 전국 확장 가능 구조 유지

## 현재 제품 상태 진단
현재 서비스는 메인 화면에서 기업 카드와 추천 일자리 카드를 탐색할 수 있지만,  
"자세히 보기" 이후 행동으로 이어지는 흐름(지원, 저장, 신청)이 약합니다.

핵심 갭:
- 탐색은 가능하지만 "확인 -> 행동" 단계가 끊김
- 장애 유형 관점 필터가 부족함
- 지원금 계산 결과가 실제 신청 행동으로 연결되지 않음

## 사용자별 주요 니즈와 갭

### 장애인 구직자
- 원하는 것:
  - 내 장애 유형/근무 조건에 맞는 기업과 공고 탐색
  - 기업의 실근무환경과 친화도 근거 확인
  - 지원금/혜택 정보 확인 후 실제 행동 연결
  - 관심 공고 저장 및 재방문
- 현재 갭:
  - 장애 유형 필터 부재
  - 회사/공고 상세 페이지 부족
  - 찜/북마크 기능 부재
  - 지원 플로우 부재

### 기업 인사담당자 (보조 사용자)
- 원하는 것:
  - 채용 시 받을 수 있는 지원금과 절차 파악
  - 장애인 채용 친화도를 개선할 수 있는 기준 확인
- 현재 갭:
  - 기업 관점 전용 흐름/페이지 부재

## 개선 우선순위
높은 임팩트 기준 우선순위:
1. 실데이터 연결 안정화
2. 회사 상세 페이지
3. 구인 상세 페이지
4. 지원 행동 연결(지원금 신청/지원 링크)
5. 사용자 프로필/로그인(후순위)

## 단계별 로드맵

### Phase 1 (2~3주): 여정 완성
목표: "탐색 -> 확인 -> 행동" 흐름을 끊김 없이 연결

- 경기도 집중 운영 반영
  - 추천/기업 목록 기본 필터를 `경기`로 설정
  - 경기도 중심 데이터 품질 점검 및 표기 정합성 우선 개선
- 실데이터 연결
  - `/jobs/live`, `/companies/standard-workplaces/live` 기반 매핑 정리
  - 프론트 타입과 백엔드 응답 필드 변환 계층 정리
- 회사 상세 페이지(`company/[id]`)
  - 친화도 점수 + 하위 지표
  - 고용 현황 수치
  - 표준사업장 인증 여부
  - 관련 채용 공고 리스트
- 구인 상세 페이지(`job/[id]`)
  - 모집기간, 급여, 요구경력/학력
  - 작업환경 필드(env*) 표시
  - "지원금 계산으로 이동" 액션 연결

### Phase 2 (1~2개월): 재방문 동기 강화
목표: 한 번 방문에서 끝나지 않는 제품 경험 구축

- 장애 유형/근무환경 필터 추가
- 지원금 계산기 -> 실제 신청 링크 연결
- 찜/북마크(우선 localStorage) 추가
- 홈 통계 수치 실데이터 동기화
- 경기도 성과 검증 후 수도권 -> 전국 확장 계획 수립

### Phase 3 (3개월+): 신뢰도/차별화 강화
목표: 추천 결과의 설득력과 장기 사용성 강화

- 매칭 점수 근거 설명 UI
- 기업 지표의 시계열 비교
- 관심 조건 기반 채용 알림

## 지금 당장 가장 가성비 높은 실행 항목
- 회사 상세 페이지 + 구인 상세 페이지 구현
  - 기존 카드 컴포넌트 재사용 가능
  - 백엔드 live API가 이미 존재
  - 사용자 여정을 "탐색 -> 확인 -> 행동"으로 빠르게 전환 가능
